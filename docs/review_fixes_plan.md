## 2. 判定の根拠（重要点）

### A. ファイアボール短縮
- 実装上、ファイア中の鉛直速度は `vy = max(bvy, FIREV)`。`bvy` は毎ステップ `+= GRAV*dt` で増加し `MAXV=1700` まで上がるため、貫通距離は `FIREV×FIRE_TIME` の概算（≈1958px≒15階）を上回り得る。
- `FIRE_TIME=0.48` で貫通距離は概ね `~1600×0.48 ≒ 770px ≒ 6階`（実測で1発あたり約5〜6階）。爽快感は維持しつつ、半分スキップの過剰さを解消。
- `MAX_FLOOR=30`（ブランド「30階」）は変更しない。

### B. クリア判定（現状維持の理由）
- 仕様 `docs/specification.md` で「30階到達＝29階の穴を通過して現在階層が30になった時点」「30階到達時に+3000」と**明示的に定義**済み。現実装はこの定義に正確に一致し、Off-by-oneではなく意図された勝利条件。
- 変更（`floor > MAX_FLOOR`）は、(1) 仕様と矛盾、(2) クリアに要する通過数が28→29個に増え**既存ランキングのスコア基準がずれる**、(3) HUD/README/仕様の改訂が必要、という副作用を伴う。出荷・マージ済みの契約を「バグ修正」名目で変えるべきでないと判断。
- 「30個の障害物を体験させたい」という設計意図に変える場合は、仕様変更として別途合意のうえ対応可能（HUD・ランキング注記も同時更新）。本修正では現状維持。

### 2A. ファイア中の穴通過ボーナス
- 「ファイア中はすべての階で `+50`」に統一。床破壊（`smashed`）でも穴通過でも同額を付与し、「無敵中はあえて床に当てる」という逆インセンティブを解消。ラベルは破壊時 `スマッシュ +50`、穴通過時 `ファイア +50`。

## 3. 実装計画（`index.html`）

1. **A**: 定数 `FIRE_TIME` を `1.45 → 0.55`。
2. **2A**: `passFloor(p,smashed)` の加点を `if(smashed || game.fire>0){ +50 }` に変更（ラベル分岐）。通常時のコンボ加点ロジックは不変。
3. **2B**: `resize()` の `view.R=Math.min(W*0.44,158)` を `Math.min(Math.min(W,H)*0.44,158)` に。`r0/RBALL` は従来通り `R` 由来。
4. **2C**: `#retireBtn` の `onclick` に `confirm('リタイアして結果画面へ進みますか？')` を挟み、キャンセル時は終了しない。
5. **3-fill**: `PAL` 各色に上面用の明色 `safeTop/dgrTop`（白16%混色）を初期化時に算出し、`drawPlatform` 上面塗りを1回に集約（強調くさびのみ追加 `fill`）。
6. ドキュメント更新: `docs/specification.md`（ファイア時間・スコア）/ `README.md` / `docs/review_checklist.md`。

## 4. リスク・検証

- いずれも局所変更で、状態機械・RPC・描画モデルの骨格は不変。
- 検証（iPhone SE 320/375・iOS UA・実ブラウザ自動プレイ）:
  - ファイア発動後の貫通階数が約5〜6階に収まる。
  - ファイア中の穴通過で `+50` が入る。
  - 横画面（例: 667×320）で塔が画面内に収まり自機が見える。
  - リタイア確認でキャンセルすると続行、OKで結果へ。
  - 30階クリア・通常時の危険帯終了・本番経路でデバッグ非公開・JSエラー0・送信1/取得1 が回帰なし。
  - `node --check` 構文OK。
